# Contributing to SUNIE / LUMIE

まず、このプロジェクトに関心を持ってくださりありがとうございます。  
SUNIE / LUMIE は、**「美・安定・最適の統合」**を軸に、  
学習・推論・意味構造を新しい形で再構築する試みです。

本プロジェクトは単なるコードの集合ではなく、  
**思想・構造・哲学を含む“概念体系”**として設計されています。  
そのため、貢献にあたっては以下の方針をご理解いただけると幸いです。

---

## 🌱 1. プロジェクトの哲学（必読）

SUNIE / LUMIE は次の原則を大切にしています。

### **1.1 美・安定・最適の三位一体**
- コードは「動けばよい」ではなく、  
  **読みやすさ・構造の美しさ・安定性**を重視します。
- 乱雑な高速化よりも、**長期的な拡張性**を優先します。

### **1.2 概念の一貫性**
- 新しい機能や改善は、  
  **プロジェクト全体の意味構造と整合性があること**を重視します。
- 「なぜそれが必要なのか」を説明できることを歓迎します。

### **1.3 自律性と創造性**
- SUNIE / LUMIE は、既存の枠組みを模倣するのではなく、  
  **新しい枠組みを創ること**を目的としています。
- そのため、単なる移植や模倣よりも、  
  **新しい視点・新しい構造の提案**を歓迎します。

---

## 🛠 2. 技術的な貢献ガイドライン

### **2.1 Issue の立て方**
- バグ報告、提案、質問などは Issue にてお願いします。
- 可能であれば以下を含めてください：
  - 再現手順
  - 期待される挙動
  - 実際の挙動
  - 関連するログやスクリーンショット

### **2.2 Pull Request の方針**
- 小さく、明確な単位での PR を歓迎します。
- PR には以下を含めてください：
  - 目的（何を、なぜ変えるのか）
  - 変更内容の概要
  - 影響範囲（他のモジュールへの影響など）
  - 必要であれば簡単なテスト

### **2.3 コーディング規約**
- 言語ごとの一般的なスタイルガイドに準拠してください。
- 関数名・変数名は意味が明確であることを重視します。
- コメントは「何をしたか」よりも  
  **「なぜそうしたか」**を書いていただけると助かります。

## 🔍 3. プロジェクト構造の理解

貢献前に、以下のドキュメントを読んでいただくとスムーズです。

- `README.md`  
- `ARCHITECTURE.md`（存在する場合）  
- 各モジュールの `docs/`  

SUNIE / LUMIE は階層構造と意味構造が密接に結びついているため、  
**全体像の理解が非常に重要**です。
